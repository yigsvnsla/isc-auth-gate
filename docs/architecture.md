# Architecture

## Stack decisions

### Better Auth over NextAuth.js

NextAuth.js v5 (Auth.js) was the initial consideration. Better Auth won on:

| Factor | Better Auth | Auth.js |
|--------|-------------|---------|
| OAuth 2.1 provider | Built-in plugin (`@better-auth/oauth-provider`) | Requires custom implementation |
| Organization/multi-tenant | First-party plugin with dynamic roles | No equivalent |
| Admin API | Full CRUD + RBAC `createAccessControl` | Manual implementation |
| OpenAPI docs | `openAPI()` plugin generates Swagger | Not available |
| Drizzle adapter | First-class, schema auto-generated | Community adapter, manual schema |
| Plugin DX | Monorepo, consistent API | Fragmented across packages |

### Bun over Node.js

Bun was chosen for:
- Single runtime for dev, test, build (`bun test`, `bun x`, `bun run`)
- ~3x faster package installs than npm
- Native TypeScript execution (no ts-node)
- Built-in test runner compatible with Jest API

### Drizzle ORM over Prisma

- Better Auth generates Drizzle schema directly from config
- SQL-first migrations (not Prisma schema DSL)
- Lighter bundle, no binary engine
- Better Auth team maintains official Drizzle adapter

### shadcn/ui base-nova preset

- `base-nova`: acromatic minimal style, no visual noise
- Dark mode via `prefers-color-scheme` CSS media query (not class-based)
- Components are local copies (edit without breaking upstream)
- Blue accent color used sparingly for emphasis (auth buttons, sidebar active state, email templates)

## Auth system design

### Authentication flow

```
Browser          Next.js Server          Better Auth          Database
  │                    │                     │                  │
  │  request           │                     │                  │
  ├───────────────────►│                     │                  │
  │                    │  session check      │                  │
  │                    ├────────────────────►│                  │
  │                    │◄──── user/session ──┤                  │
  │                    │                     │                  │
  │  /dashboard        │                     │                  │
  │◄───────────────────┤                     │                  │
```

The dashboard layout (`app/dashboard/layout.tsx`) checks session server-side. Unauthenticated requests redirect to `/auth/sign-in`.

### Plugin architecture

```
betterAuth({ plugins: [
  adminPlugin({...}),         # Admin API: user CRUD, ban, role management
  organization({...}),        # Multi-tenant: orgs, members, dynamic roles
  openAPI(),                  # Swagger at /api/auth/reference
  nextCookies(),              # Cookie serialization for server components
  jwt(),                      # JWT token support (required by oauthProvider)
  oauthProvider({...}),       # OAuth 2.1 server: authorize, token, userinfo
]})
```

### RBAC model

```
createAccessControl(statements)
      │
      ├── static roles: admin, moderator, user
      │   └── passed to adminPlugin()
      │
      └── org roles: orgSystemAdmin → expanded via dynamicAccessControl
          └── passed to organization()
```

Roles are defined once in `lib/permissions.ts` and reused across server config, client config, and org plugin.

## Ponytail convention

`// ponytail:` comments mark intentional shortcuts with documented upgrade path. These are not TODOs — they are conscious trade-offs that could be expanded when requirements grow.

Examples in codebase:

| File | Mark | Rationale |
|------|------|-----------|
| `lib/auth.ts` | `scopes: ["openid", "profile", "email", "offline_access"]` | Minimal OIDC scopes. Add custom resource scopes when defined |
| `lib/auth.ts` | `cachedTrustedClients: new Set(["isc-gate-dashboard"])` | Only dashboard is trusted. Expand with MCP agent clients |
| `lib/permissions.ts` | `organization: ["update", "delete"]` | Org-native statements needed by DAC. Added alongside defaultStatements |

## Security

- Env vars validated by Zod at startup — missing vars crash immediately, not at runtime
- CSRF check enabled (`disableCSRFCheck: false`)
- Origin check enabled (`disableOriginCheck: false`)
- Secure cookies in production (`useSecureCookies: true`)
- Rate limiting: 10 requests / 60s window
- Email verification required for login
- OAuth PKCE supported per client
- PostgreSQL SSL configurable per environment

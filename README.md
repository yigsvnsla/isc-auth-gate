# isc-auth-gate

Authentication gateway built on Next.js 16, Better Auth, Drizzle ORM, and PostgreSQL.

## Stack

| Layer | Choice |
|-------|--------|
| Runtime | [Bun](https://bun.sh) (not Node) — `bun test`, `bun x`, `bun run` |
| Framework | [Next.js 16](https://nextjs.org) + React 19, React Compiler enabled |
| Auth | [Better Auth](https://better-auth.com) with Microsoft OAuth, RBAC, admin API, orgs |
| Database | PostgreSQL via [Drizzle ORM](https://orm.drizzle.team), managed with Drizzle Kit |
| UI | [shadcn/ui](https://ui.shadcn.com) base-nova preset, Tailwind CSS 4 |
| Email | [React Email](https://react.email) with [nodemailer](https://nodemailer.com) (SMTP) |
| State | [SWR](https://swr.vercel.app) (server data), [Jotai](https://jotai.org) (client atoms) |

## Quick start

```bash
bun install
cp .env.example .env        # edit with your values
bun run database:push       # push schema to PostgreSQL
bun run seed:run            # seed test data
bun run dev                 # http://localhost:3000
```

## Commands

```bash
bun run dev             # dev server (:3000)
bun run build           # production build
bun start               # start production server
bun test                # run all tests
bun test --watch        # watch mode
bun run lint            # ESLint

# Database
bun run database:push    # push schema (no migration file)
bun run database:generate  # generate migration files
bun run database:up      # apply pending migrations
bun run database:check   # validate schema
bun run database:studio  # Drizzle Studio (:4001)

# Auth schema
bun run auth:generate   # generate database/schema.ts from lib/auth.ts

# Seed
bun run seed:run         # insert test data
bun run seed:reset       # truncate seeded tables

# Email preview
bun run email:dev        # React Email preview (:3001)
```

## Auth flows

- **Email/password** — register → verify email → login → session
- **Microsoft OAuth** — redirect → Entra ID login → callback → session
- **OAuth 2.1 Provider** — 3rd-party apps can authenticate via this server (clients managed at `/dashboard/administration/oauth`)
- **Organizations** — multi-tenant with dynamic roles and per-org permissions

All auth endpoints served at `/api/auth/[...all]` via Better Auth's `toNextJsHandler`.

Swagger/OpenAPI reference: `/api/auth/reference` (dev only).

## Environment variables

See `.env.example` for all required vars. Validated by Zod in `env/index.ts` — never access `process.env` directly.

### Required

| Var | Description |
|-----|-------------|
| `BETTER_AUTH_SERVER_SECRET` | Signing key for auth tokens |
| `BETTER_AUTH_MICROSOFT_CLIENT_ID` | Entra ID (Azure AD) app client ID |
| `BETTER_AUTH_MICROSOFT_CLIENT_SECRET` | Entra ID client secret |
| `BETTER_AUTH_DATABASE_HOST` | PostgreSQL host |
| `BETTER_AUTH_DATABASE_NAME` | Database name |
| `BETTER_AUTH_DATABASE_USER` | DB user |
| `BETTER_AUTH_DATABASE_PASS` | DB password |
| `BETTER_AUTH_SMTP_TRANSPORTER_HOST` | SMTP server |
| `BETTER_AUTH_SMTP_TRANSPORTER_USER` | SMTP user |
| `BETTER_AUTH_SMTP_TRANSPORTER_PASS` | SMTP password |

### Gotchas

- `database/schema.ts` is **generated** by `bun run auth:generate` — do not edit manually
- `BETTER_AUTH_SERVER_TRUSTED_ORIGINS` is comma-separated in `.env`, Zod parses it to array
- React Compiler enabled — be aware of [rules](https://react.dev/learn/react-compiler#rules)
- Tests share the same database; `cleanupTestDb()` truncates tables between runs
- `@better-auth/oauth-provider` pinned to `1.6.12` (compatible with `better-auth@1.6.11`)

## Tests

```bash
bun test                    # all tests

```

Test DB connection in `tests/database.ts`. Uses `better-auth` `testUtils` plugin for isolated auth instances.

## Database

14 tables managed by Better Auth + OAuth Provider plugin:

- Users, sessions, accounts, verifications (core auth)
- Organizations, members, invitations, organization_roles (org plugin)
- OAuth clients, access tokens, refresh tokens, consents (OAuth provider)
- JWKS (key storage)

Roles are enforced via `createAccessControl` in `lib/permissions.ts` with 4 roles: `admin`, `moderator`, `user`, `orgSystemAdmin`.
